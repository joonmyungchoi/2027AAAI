# 템플릿 변인 행렬 v1 — instance 생성 설계

> 작성: 2026-06-11. 전제: `gt_design_v1.md` (GT 규칙 합성), handoff v5 §3·§4 (그리드·공통 스트림).
> 원칙: 타임라인 골격은 고정, 변인은 정적 요소 + 이벤트 슬롯 채우기로만 변주 → env 루프 구현 단일형 유지, GT 자동 합성.

---

## 1. 골격 (고정)

- 관찰 창 90분 (T−90 → T_appt), 그리드 15분 = **7 trigger** + 동적 이벤트 0~3개 즉시 trigger.
- 약속 시각 T_appt 고정 (17:00), 시작 위치·로그는 persona별 고정 재사용.
- 모든 persona(4)가 같은 이벤트 스트림 — instance당 4 persona-run (공통 스트림 원칙 유지).

---

## 2. 변인 행렬

### 정적 변인 (시나리오 골격)

| 변인 | 레벨 | 시험 대상 |
|---|---|---|
| S1 목적지 익숙도 | 익숙(로그 방문 N회) / 낯섦(0회) — 2 | CASE III 익숙도 마스크 |
| S2 이동 구성 | 도보 근거리(~15분) / 도보+지하철(~30분) / KTX 장거리(~60분) — 3 | travel 함수, 날씨 민감 구간(도보 비중) |

### 이벤트 슬롯 변인 (동적)

| 변인 | 레벨 | 시험 대상 |
|---|---|---|
| E1 날씨 | 없음 / 비-발사창 전 / 비-발사창 후(재알림) / 위험기상(1층 강제) — 4 | CASE IV, 승격·강제 경계 |
| E2 상대 | 없음 / 지연+10 / 지연+60 / 취소 / 장소변경 — 5 | CASE VI, override 1·3 |
| E3 교통 | 없음 / 지연+5(미세) / 지연+20(승격) — 3 | **승격 규칙 경계 양측** (gt_design §1) |
| E4 user state | 없음(폰 사용) / 통화-발사창 겹침 / 회의-겹침 / 운전-겹침 — 4 | CASE V 채널맵 |

- 전체 조합 = 2×3×4×5×3×4 = **1,440 셀**. 전수 생성하지 않고 §3 샘플링.
- 슬롯 시각은 레벨별 기본값 + ±15분 지터 (그리드 정렬 유지).

### 제약 (조합 금지 규칙)

- E2=취소 이후 E1·E3는 GT에 무효 (override 1) — 취소 instance에선 E1·E3=없음으로 고정 (낭비 방지).
- E4 겹침 레벨은 합성된 발사창 기준으로 배치 — GT 합성 후 발사창과 겹치도록 시각 조정.
- 운전 state는 S2에 도보 구간이 있는 레벨과만 결합 금지 대신 "이동 전 운전(외출 중)" 으로 정의.

## 3. 샘플링 설계 — 1차 100 instance (확정), 검증 후 확장

| 구성 | 수 | 방법 |
|---|---|---|
| Core | 90 | 층화 샘플링 — 각 변인 레벨 균형 + **모든 레벨 쌍 ≥2회** (pairwise coverage) |
| No-op | 10 (~10%) | 약속 없음 / 이미 완료 / 당일 아님 — **전 trigger silent가 GT**. restraint 시험 + "전부 fire" 게이밍 차단 |
| 1차 합계 | **100** | × 4 persona = 400 persona-run, 그리드 7 + 이벤트 0~3 = **총 ~3,500 판정** |

- **확장 계획 (확정, 2026-06-11)**: 1차 100개로 프로세스(생성기→GT 합성→검수→평가 파이프라인) 검증 후, 같은 생성기로 150~200까지 확장 (쌍 커버리지 ≥3회로 상향). 생성기·시드 기반이라 확장 비용은 샘플링 재실행 + 검수 표본 추가뿐.
- 시드 고정, 생성 스크립트로 재현 가능하게.

### GT 모호성 가드

- 합성 후 fire_time이 band 경계 ±2.5분 이내면 → 해당 instance의 이벤트 시각/Δ를 5분 조정 후 재합성. (경계 걸친 GT는 채점 분쟁 소지 — E3의 +5/+20처럼 *의도된* 경계 시험만 남김)

## 4. 로그 연결 규칙

- 로그는 **persona당 1벌(clean/noise 2버전) 고정 재사용** — instance마다 새 로그 생성 안 함 (KnowU 방식, 비용·일관성).
- S1 익숙도는 로그의 **목적지 방문 기록으로 실현**: 로그에 "자주 가는 장소 풀"(각 N회 방문)을 심고, 템플릿이 목적지를 익숙 풀/낯섦 풀에서 선택.
- instance 날짜는 로그 기간 이후로 설정 (시간 정합).
- 빈번도·정도·lead 신호는 instance 무관 로그 자체 속성 (gt_design §6 open — 행동화 설계와 함께 확정).

## 5. 평가 비용과 GUI subset

- GUI 전수 평가는 비현실적 (모델당 수천 에피소드). **2단 구성 확정 (2026-06-11)**.

| 트랙 | 범위 (1차) | 모드 | 역할 |
|---|---|---|---|
| Full | 100 × 4 | **text-state** (state·이벤트를 텍스트로 주입, MCP mock 동일) | 메인 리더보드 |
| GUI | 층화 25~30 × 4 | 에뮬레이터 (CASE V state를 GUI에서 읽음) | GUI grounding 효과 입증 + design_review P1-7 ablation 겸용 |

- text-state와 GUI 점수 상관을 보고하면 "text-state가 유효한 프록시"라는 방어까지 확보.
- GUI subset은 E4(state) 레벨이 고르게 들어가도록 층화. 확장 시 비례 확대.
