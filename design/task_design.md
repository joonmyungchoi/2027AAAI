## SETTING I — Research Question

**기존 연구**: 어떤 도움을 줄 것인가를 고민
**우리 연구**: Proactive Agent를 *언제 & 어떻게* 접근해야 사용자가 거슬리지 않게 받아들일 것인가?

### 고려가 필요한 지표
- **긴급도**: 시간이 얼마나 남았는가 + 상황
- **유저의 과거 행동 패턴 (H)**: 이 사람은 알림에 어떻게 반응해왔는가?
- **일정의 중요도**: 이 일정이 얼마나 중요한가?
- **유저의 현재 상태**: 알림을 받을 수 있는 상태인가? (활동 중, 통화 중, 수면 중)
- **유저의 개입 선호도 (H)**

기준 상황 예시 → 오후 4시 강남 약속

## Difficulty Levels (LV1–LV5)

각 레벨은 *agent가 트리거 결정을 위해 통합해야 하는 정보원의 수*를 늘려가며 난이도를 올림.

### LV1. 명시적 정보

```
상황: 2시 온라인 미팅 예정

필요한 정보:
  캘린더 (미팅 시간)

언제 트리거 (✓):
  1시 50분 (10분 전)
  단, 미팅 접속 중 → 트리거 X

어떻게 (고정 규칙):
  유저가 다른 앱 사용 중 →  알림
  유저가 폰을 안 보는중 → 알림 + 진동
```

### LV2. 명시적 정보 + 과거 패턴

```
상황: 4시 강남 약속 (가본 곳)

필요한 정보:
  캘린더 (약속 시간)
  H (과거 이동 패턴)

언제 트리거:
  H 확인 → 소요시간 파악(1시간) → 2시 50분
  단, 이미 이동 중 → 트리거 X

어떻게:
  유저가 활동 중 → 알림 + 출발 제안
  유저가 수면 중 → 알림 + 알람
```

### LV3. 명시적 정보 + 현재 환경

```
상황: 4시 강남 약속 (안 가본 곳)

필요한 정보:
  캘린더 (약속 시간, 장소)
  지도 검색 (현재 위치 → 목적지 소요시간)

언제 트리거:
  지도 검색 → 소요시간 파악(1시간) → 역산 → 2시 50분 알람
  단, 이미 이동 중 → 트리거 X

어떻게:
  유저가 활동 중 → 지도 검색(소요시간파악) + 팝업 알림 예약
  유저가 통화 중 → 지도 검색(소요시간파악) + 팝업 알림 + 진동 예약
  유저가 수면 중 → 지도 검색(소요시간파악) + 팝업 알림 + 알람 예약
```

### LV4. 명시적 정보 + 과거 패턴 + 현재 환경

```
상황: 4시 강남 약속 (가본 곳 + 비가 옴)

필요한 정보:
  캘린더 (약속 시간)
  H (과거 이동 패턴)
  현재 날씨 (비 → 소요시간 증가)

언제 트리거:
  H + 날씨 결합 → 소요시간 재계산 → 2시 30분
  단, 이미 이동 중 → 트리거 X

어떻게:
  유저가 활동 중 → 지도 검색(날씨 반영 소요시간 파악) + 알림 + 출발 제안
  유저가 수면 중 → 지도 검색(날씨 반영 소요시간 파악) + 알림 + 알람
```

### LV5. 명시적 정보 + 과거 패턴 + 현재 환경 + 실시간 변동

```
상황: 4시 강남 약속 + 이동 중
     대중교통 사고 발생

필요한 정보:
  캘린더 (약속 시간)
  H (과거 이동 패턴)
  실시간 교통 상황 (사고 발생)
  메시지 앱 (친구 연락처)

언제 트리거:
  사고 감지 즉시 → 소요시간 재계산
  단, 이미 목적지 도착 → 트리거 X
  단, 이미 메시지 보냄 → 트리거 X

어떻게:
  유저가 활동 중 → 사고 확인 + 알림 + 지연 메시지 추천
  유저가 통화 중 → 사고 확인 + 알림 + 진동 + 지연 메시지 추천
  유저가 수면 중 → 사고 확인 + 알림 + 알람 + 지연 메시지 추천
```

## Evaluation Rubric — action → consideration

평가의 핵심: agent가 *결과 행동* 만이 아니라 *그 결정에 이르기까지 고려한 요소* 까지 평가.

### Setting (예시 LV4)

```
상황: 4시 강남 약속 (가본 곳 + 비가 옴)

필요한 정보:
  캘린더 (약속 시간)
  H (과거 이동 패턴)
  현재 날씨 (비 → 소요시간 증가)

언제 트리거:
  H + 날씨 결합 → 소요시간 재계산 → 2시 30분
  단, 이미 이동 중 → 트리거 X

어떻게:
  유저가 활동 중 → 지도 검색(날씨 반영) + 알림 + 출발 제안
  유저가 수면 중 → 지도 검색(날씨 반영) + 알림 + 알람
```

### M2. WHEN — Triggering (timing)

```
필수 고려 요소:
  F1: 캘린더 약속 시간 확인
  F2: H에서 과거 소요시간 확인
  F3: 현재 날씨 확인
  F4: 날씨 반영해서 소요시간 재계산

정답 트리거 구간: 2시 10분 ~ 2시 30분
(소요시간 1시간 30분 + 준비 10분 역산)

CASE A:
  "캘린더 확인 → 4시 강남 약속 (F1 ✅)
   과거 기록 보니 강남까지 보통 1시간 (F2 ✅)
   현재 날씨 비 → 소요시간 증가 예상 (F3 ✅)
   1시간 30분으로 재계산 → 2시 20분 출발 (F4 ✅)"
   → 트리거 시점: 2시 20분

  Factor Coverage: 4/4 = 1.0
  Result Accuracy: 구간 내 = 1.0

CASE B:
  "캘린더 확인 → 4시 강남 약속 (F1 ✅)
   과거 기록 보니 강남까지 1시간 (F2 ✅)
   2시 50분에 출발하면 됨"
   → 날씨 미고려 (F3 ❌, F4 ❌)
   → 트리거 시점: 2시 50분

  Factor Coverage: 2/4 = 0.5
  Result Accuracy: ±10분 이내 = 0.5 (구간 밖이지만 가까움)

CASE C:
  "4시에 약속이 있네 (F1 ✅)
   3시 30분에 알려줘야지"
   → H, 날씨 모두 미고려 (F2 ❌, F3 ❌, F4 ❌)
```

### M3. HOW — Delivery (intervention depth/style)

```
필수 고려 요소:
  G1: 유저의 현재 상태 확인
  G2: 유저의 과거 반응 패턴 (H)
  G3: 일정의 중요도 판단
  G4: 유저의 개입 선호도 (H)

정답 개입 깊이: 알림 + 경로 제안
(활동 중 + 반응 빠름 + 일반 미팅 + 제안까지 수용)

CASE A (정답):
  "유저 현재 활동 중 (G1 ✅)
   H 보니 알림에 빠르게 반응함 (G2 ✅)
   강남 약속, 일반 미팅 수준 (G3 ✅)
   H 보니 제안까지는 수용하는 편 (G4 ✅)
   → 알림 + 경로 제안"

  Factor Coverage: 4/4 = 1.0
  Result Accuracy: 정답과 일치 = 1.0

CASE B (과도):
  "유저 현재 활동 중 (G1 ✅)
   비 오니까 택시 예약해야겠다
   → 택시 예약까지 실행"
   → G2 ❌ G3 ❌ G4 ❌

  Factor Coverage: 1/4 = 0.25
  Result Accuracy: 한 단계 과도 = 0.5

CASE C (부족):
  "알림 보내야지
   → 팝업만"
   → G1 ❌ G2 ❌ G3 ❌ G4 ❌

  Factor Coverage: 0/4 = 0.0
  Result Accuracy: 한 단계 부족 = 0.5
```
